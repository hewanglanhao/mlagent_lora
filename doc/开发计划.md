重点不是提交一个固定 kernel，而是让系统在 `run.sh` 启动后自动经历：生成代码 → 编译 → 正确性测试 → benchmark → profiling → 分析瓶颈 → 继续生成/变异候选代码 → 保存当前最优 `optimized_lora.cu` 的闭环。作业明确要求最终优化目标是：

[
Y = WX + A(B^T X), \quad r = 16,\quad d \in [3584, 4608]
]

并且系统运行期间必须始终维护一个可编译的 `optimized_lora.cu`，最终评测会读取这个文件；正确性是硬门槛，性能和 agent 工程质量共同评分。

---

# 1. 总体架构

推荐采用**Supervisor + 多专职 Agent + 实验数据库**的架构。

```text
run.sh
  |
  v
Agent Orchestrator / Supervisor
  |
  +-- Constraint & Spec Manager
  +-- Environment Inspector Agent
  +-- Search Space Planner Agent
  +-- CUDA Candidate Generator Agent
  +-- Static Code Review Agent
  +-- Build & Compile Agent
  +-- Correctness Test Agent
  +-- Benchmark Agent
  +-- Profiling Agent
  +-- Performance Diagnosis Agent
  +-- Optimization Mutation Agent
  +-- Best Candidate Manager
  +-- Experiment Memory / Log Store
```

核心思想是：
**Supervisor 不直接写最终代码，而是调度多个 agent 反复做实验。每一次实验都产生一个 candidate CUDA 文件，只有通过编译、正确性和 benchmark 的候选才有资格更新 `optimized_lora.cu`。**

---

# 2. 系统运行主循环

系统可以设计成如下闭环：

```text
初始化
  |
  |-- 生成第一个保底正确版本 optimized_lora.cu
  |-- 读取 GPU / CUDA / PyTorch / nvcc 环境
  |-- 生成 synthetic test cases: d ∈ [3584, 4608]
  |
循环直到时间耗尽或收敛:
  |
  |-- 生成候选 CUDA 实现 candidate_i.cu
  |-- 静态检查接口、依赖、shape 泛化能力
  |-- 编译 candidate_i.cu
  |-- 正确性测试: 对比 PyTorch reference
  |-- benchmark: 多个 d，多轮 warmup + median latency
  |-- profiling: Nsight / nvprof / torch profiler / CUDA events
  |-- 诊断瓶颈
  |-- 记录实验结果
  |-- 若更优，则复制为 optimized_lora.cu
  |-- 根据瓶颈生成下一轮优化方向
```

作业要求 agent 要能生成候选实现、编译测试、benchmark、比较方案、迭代改进，并始终保留当前 best `optimized_lora.cu`；hidden tensor 不可见，所以本地搜索必须用 public range 内的 synthetic tensor。

# 3. 每个模块 / Agent 的功能设计

## 3.1 `run.sh` 启动模块

这是评测系统唯一直接调用的入口。

**功能：**

1. 设置环境变量，例如：

   * `PYTHONUNBUFFERED=1`
   * `MAX_OPT_TIME=1700`，留出清理时间，避免 30 分钟硬超时。
   * 可选：`OPENAI_API_KEY` 或本地 LLM 配置。
2. 启动主控脚本，例如：

   * `python3 agent_main.py`
3. 结束前保证：

   * 根目录下存在 `optimized_lora.cu`
   * 这个文件是当前 best 的可编译版本
   * 即使中途失败，也不能删除 best 文件

**设计重点：**

`run.sh` 不负责优化逻辑，只负责启动和兜底。
如果 agent crash，`run.sh` 应该保留已有 `optimized_lora.cu`，而不是让提交目录处于无效状态。

---

## 3.2 Agent Orchestrator / Supervisor

这是系统大脑，负责调度所有 agent。

**功能：**

1. 管理全局时间预算。
2. 维护当前状态：

   * 当前 best latency
   * 当前 best correctness error
   * 当前 best CUDA 文件路径
   * 已尝试过的优化策略
   * 编译失败原因
   * profiling 结果
3. 决定下一步做什么：

   * 继续探索新 kernel family
   * 微调 tile size
   * 改 block/thread 配置
   * 尝试 fusion
   * 回退到上一个稳定版本
4. 控制风险：

   * 不让错误候选覆盖 `optimized_lora.cu`
   * 每轮实验有 timeout
   * 超时前提前停止，保存 best

**输入：**

* 作业 spec
* 历史实验结果
* profiling 诊断
* 当前时间预算

**输出：**

* 下一轮 candidate 生成任务
* 是否更新 best
* 最终 `optimized_lora.cu`

---

## 3.3 Constraint & Spec Manager

这个模块专门维护作业约束，防止 LLM 生成不符合要求的方案。

**功能：**

1. 固化 operator 定义：

   [
   Y = WX + A(B^T X)
   ]

2. 固化 shape：

   * `W: d x d`
   * `X: d x d`
   * `A: d x 16`
   * `B: d x 16`
   * `d ∈ [3584, 4608]`
   * float32

3. 固化输出接口：

```cpp
torch::Tensor forward(torch::Tensor W,
                      torch::Tensor X,
                      torch::Tensor A,
                      torch::Tensor B);
```

4. 固化最终文件约束：

   * 最终必须是单文件 `optimized_lora.cu`
   * 必须有 `PYBIND11_MODULE`
   * 不能依赖额外 `.cu/.cuh/.h/.cpp`
   * 不能只提交静态手写 kernel
   * 不能 hardcode hidden shape 或数据

5. 提供“约束检查清单”给 Static Code Review Agent。

## 3.4 Environment Inspector Agent

这个 agent 负责读取当前硬件和软件环境。

**功能：**

1. 查询 GPU：

   * 名称：预期 RTX 3090
   * SM 数量
   * shared memory 限制
   * max threads per block
   * warp size
   * memory bandwidth 粗略估计
   * compute capability，RTX 3090 通常是 sm_86

2. 查询编译环境：

   * `nvcc --version`
   * PyTorch version
   * CUDA version
   * GCC version

3. 生成硬件参数配置：

   * 推荐 block size
   * candidate tile size 范围
   * 是否启用 `cp.async`，如果实现复杂可降级
   * 是否使用 Tensor Core，float32 场景通常主要考虑 TF32/cuBLAS 或常规 FP32

**输出：**

```json
{
  "gpu": "RTX 3090",
  "sm": 86,
  "max_threads_per_block": 1024,
  "shared_mem_per_block": "...",
  "candidate_tile_m": [16, 32, 64],
  "candidate_tile_n": [16, 32, 64],
  "candidate_tile_k": [8, 16, 32]
}
```

## 3.5 Synthetic Data Generator

官方 hidden input 不可见，所以 agent 必须自己生成测试数据。

**功能：**

1. 在 public range 内生成多个 `d`：

   * 小：3584
   * 中：4096
   * 大：4608
   * 随机：例如 3712、3840、4352、4480
2. 生成 float32 CUDA tensor：

   * `W = torch.randn(d, d, device="cuda")`
   * `X = torch.randn(d, d, device="cuda")`
   * `A = torch.randn(d, 16, device="cuda")`
   * `B = torch.randn(d, 16, device="cuda")`
3. 控制随机种子，保证实验可复现。
4. 可生成不同数值尺度的数据：

   * 标准正态
   * 小幅值随机数
   * 稍大幅值随机数
     用于发现数值误差问题。

**输出：**

* 本地 benchmark dataset
* correctness dataset
* stress dataset

## 3.6 Baseline Candidate Generator

这个模块负责在一开始快速生成一个**保底可编译、正确性优先**的 `optimized_lora.cu`。

**功能：**

1. 在 agent 开始优化前，立即生成一个 baseline 版本。
2. 该版本可以直接调用 PyTorch/ATen 的矩阵乘法实现：

   * `W @ X`
   * `B.t() @ X`
   * `A @ tmp`
   * 相加
3. baseline 可能不快，但必须：

   * 正确
   * 可编译
   * 接口符合要求
   * 单文件
4. 写入 `optimized_lora.cu`，作为初始 best。

**为什么需要它：**

作业明确要求不要等到最后才产出第一个工作版本，系统运行期间必须始终维护最新可编译版本。

## 3.7 Search Space Planner Agent

这个 agent 负责决定系统要探索哪些优化方向。

针对 LoRA 算子：

[
Y = WX + A(B^T X)
]

可以拆成：

1. 主项：`WX`，大小是 `d x d` 乘 `d x d`
2. 低秩项：

   * `T = B^T X`，大小是 `16 x d`
   * `R = A T`，大小是 `d x d`
3. 输出：

   * `Y = WX + R`

由于 `r = 16` 很小，低秩项有特殊优化空间。

**候选优化方向：**

### 方向 A：ATen / cuBLAS 组合型

先生成可靠版本：

```text
Y0 = matmul(W, X)
T  = matmul(B^T, X)
R  = matmul(A, T)
Y  = Y0 + R
```

优点：

* 正确性高
* 利用成熟 GEMM
* 实现风险低

缺点：

* kernel launch 多
* 中间 tensor 多
* 不一定比 PyTorch reference 快

---

### 方向 B：手写低秩项 kernel + cuBLAS 主 GEMM

主项 `WX` 仍交给高性能 GEMM，低秩项 `A(B^T X)` 手写优化。

流程：

```text
Y = W @ X                         # 高性能 GEMM
T = B^T @ X                       # 16 x d，小 r GEMM
Y += A @ T                        # rank-16 update，手写 CUDA kernel
```

重点优化 `Y += A @ T`：

[
Y_{ij} += \sum_{k=0}^{15} A_{ik}T_{kj}
]

这个 kernel 每个输出元素只需要 16 次 FMA，非常适合手写。

优点：

* 低秩项结构固定，容易优化
* 可减少一部分 PyTorch overhead
* 正确性相对好控

---

### 方向 C：融合低秩两步

直接计算：

[
A(B^T X)*{ij} = \sum*{k=0}^{15} A_{ik} \left(\sum_{p=0}^{d-1} B_{p k} X_{p j}\right)
]

理论上可以 fuse，但实际会导致每个 `Y_ij` 重复计算 `B^T X`，计算量暴涨，不推荐直接完全融合。

更合理的是：

```text
Step 1: T = B^T X       # 16 x d
Step 2: Y += A T        # d x d
```

即只 fusion 第二步的加法，而不是完全消除 `T`。

---

### 方向 D：多 shape 自适应 kernel

由于 `d` 不固定，candidate generator 可以生成支持 runtime shape 的 kernel：

```cpp
int d = W.size(0);
```

然后根据 `d` 选择不同 block 配置：

```text
if d <= 3840: use config A
elif d <= 4352: use config B
else: use config C
```

注意这里不能 hardcode hidden test case，只能按区间做泛化。

## 3.8 CUDA Candidate Generator Agent

这个 agent 负责生成候选 `candidate_i.cu`。

**输入：**

* 当前优化策略
* 上一轮失败/成功记录
* profiling 分析
* 作业约束
* 当前 best 代码摘要

**输出：**

* 一个新的 candidate CUDA 文件
* candidate metadata

**候选代码类型：**

1. `baseline_aten_candidate`
2. `cublas_plus_custom_rank16_candidate`
3. `custom_rank16_add_kernel_candidate`
4. `different_tile_size_candidate`
5. `different_vectorization_candidate`
6. `unroll_r16_candidate`
7. `shared_memory_T_candidate`
8. `shape_dispatch_candidate`

**必须避免：**

* 生成不符合 `forward(W, X, A, B)` 的接口
* 依赖额外头文件
* 把最终代码硬编码成一个固定模板
* 只针对 `d=4096` 优化
* 生成不可编译的 C++/CUDA 混合结构

## 3.9 Static Code Review Agent

这个 agent 在编译前先做静态检查，减少浪费时间。

**检查内容：**

1. 文件结构：

   * 是否包含 `#include <torch/extension.h>`
   * 是否有 `PYBIND11_MODULE`
   * 是否暴露 `forward`
   * 是否单文件自包含

2. 接口：

   * 参数是否是 4 个 `torch::Tensor`
   * 返回值是否是 `torch::Tensor`

3. shape：

   * 是否从 tensor 动态读取 `d`
   * 是否假设固定 `d`
   * 是否假设固定 stride 但没有检查 contiguous

4. dtype：

   * 是否假设 float32
   * 是否使用 `data_ptr<float>()`

5. CUDA safety：

   * 是否检查 tensor 在 CUDA 上
   * block/grid 是否覆盖边界
   * 是否有 out-of-bounds 访问
   * shared memory 是否超限

6. 作业合规：

   * 是否依赖外部 `.h/.cuh/.cu`
   * 是否写死最终答案
   * 是否缺少 fallback

**输出：**

```json
{
  "pass": true,
  "warnings": [
    "No explicit contiguous check",
    "Assumes d multiple of 16"
  ],
  "risk_level": "medium"
}
```

高风险 candidate 不进入编译阶段。

## 3.10 Build & Compile Agent

这个 agent 负责编译 candidate。

**功能：**

1. 使用和官方 harness 接近的方式编译：

```python
torch.utils.cpp_extension.load(
    name=f"candidate_ext_{i}",
    sources=[candidate_path],
    extra_cuda_cflags=["-O3"],
    with_cuda=True
)
```

2. 捕获编译日志：

   * nvcc error
   * C++ template error
   * missing symbol
   * linking error
   * pybind error

3. 将错误归类：

   * 语法错误
   * 接口错误
   * CUDA API 使用错误
   * PyTorch extension 错误
   * 编译超时

4. 把错误摘要传给 Code Generator / Mutation Agent。

**输出：**

```json
{
  "compiled": false,
  "error_type": "pybind_signature_error",
  "error_summary": "...",
  "compile_time_sec": 12.4
}
```

## 3.11 Correctness Test Agent

正确性是硬门槛，所以这个 agent 必须非常严格。

**功能：**

1. 对多个 shape 测试：

```text
d = 3584
d = 3840
d = 4096
d = 4352
d = 4608
```

2. 与 PyTorch reference 对比：

[
Y_{ref} = W @ X + A @ (B^T @ X)
]

3. 使用官方容差：

```python
torch.allclose(y_student, y_ref, rtol=1e-4, atol=1e-4)
```

4. 记录：

   * `passed`
   * `max_abs_err`
   * `rel_l2_err`
   * 是否出现 NaN/Inf
   * 哪个 shape 失败

5. 对边界 shape 重点测试：

   * 3584
   * 4608
   * 非 128 整除的 d，如果 public range 允许 agent 随机生成

**输出：**

```json
{
  "correct": true,
  "max_abs_err": 7.2e-5,
  "rel_l2_err": 1.1e-6,
  "failed_shapes": []
}
```

只有正确性通过的 candidate 才能进入 benchmark。

## 3.12 Benchmark Agent

这个 agent 负责性能测试。

**功能：**

1. 模拟官方 benchmark：

   * warmup
   * CUDA events
   * repeated iterations
   * median latency

2. 对多个 shape benchmark：

```text
d = 3584, 4096, 4608
```

3. 同时测：

   * candidate runtime
   * PyTorch reference runtime
   * speedup

4. 计算综合 score：

```text
score = weighted_median_speedup
```

可设置权重：

```text
3584: 0.3
4096: 0.4
4608: 0.3
```

5. 记录稳定性：

   * median
   * p25/p75
   * min/max
   * 是否有 outlier

**输出：**

```json
{
  "candidate_id": 17,
  "correct": true,
  "latency_ms": {
    "3584": 12.3,
    "4096": 16.9,
    "4608": 23.4
  },
  "torch_ms": {
    "3584": 13.5,
    "4096": 18.1,
    "4608": 25.0
  },
  "speedup": {
    "3584": 1.10,
    "4096": 1.07,
    "4608": 1.06
  },
  "aggregate_speedup": 1.075
}
```

## 3.13 Profiling Agent

这个 agent 负责找瓶颈。

**工具可以分层：**

### 轻量 profiling

用 CUDA event 分段计时：

```text
time_WX
time_BT_X
time_rank16_update
time_add
total_time
```

适合每轮都跑。

### 中量 profiling

使用 PyTorch profiler 或 CUDA synchronization 统计 kernel 时间。

### 重量 profiling

可选使用：

```bash
ncu
nsys
nvprof
```

但 30 分钟预算有限，建议只对 top candidates 做。

**关注指标：**

1. kernel launch 数量
2. global memory bandwidth
3. occupancy
4. shared memory 使用
5. register pressure
6. achieved FLOPS
7. warp divergence
8. L2 hit rate
9. 主耗时是否在 `WX`
10. 低秩项是否值得继续优化

**输出：**

```json
{
  "bottleneck": "rank16_update_memory_bandwidth",
  "evidence": {
    "rank16_update_ms": 3.1,
    "global_load_efficiency": "low",
    "occupancy": "medium"
  },
  "suggested_actions": [
    "vectorize stores with float4",
    "unroll r=16 loop",
    "increase tile_n",
    "cache A row in registers"
  ]
}
```

## 3.14 Performance Diagnosis Agent

这个 agent 把 profiling 结果转成优化建议。

**例子 1：编译通过但比 PyTorch 慢**

诊断：

```text
原因可能是 custom GEMM 不如 cuBLAS。
下一轮不要手写 WX，改为只优化低秩项。
```

建议：

```text
Use ATen/cuBLAS for W @ X.
Only custom optimize A @ (B^T X).
```

---

**例子 2：rank16 update 慢**

诊断：

```text
Y += A @ T 的计算量小，但写 Y 是 d*d 大量 global memory store。
瓶颈可能是 memory bandwidth。
```

建议：

```text
每个 thread 计算多个 contiguous columns。
使用 float4 vectorized load/store。
r=16 完全 unroll。
A[i, 0:16] 放寄存器。
T[k, j:j+vec] coalesced load。
```

---

**例子 3：小 d 快，大 d 慢**

诊断：

```text
tile 配置对大矩阵 occupancy 不够。
```

建议：

```text
为不同 d 区间做 runtime dispatch。
```

---

## 3.15 Optimization Mutation Agent

这个 agent 根据诊断结果生成下一轮改动，而不是随机乱试。

**功能：**

1. 参数变异：

   * blockDim = 128 / 256 / 512
   * tile columns = 1 / 2 / 4
   * unroll factor
   * grid mapping

2. 结构变异：

   * 是否分离 `T = B^T X`
   * 是否 fuse `Y += A @ T`
   * 是否使用 ATen matmul
   * 是否手写 rank16 kernel

3. 编译选项变异：

   * `-O3`
   * `--use_fast_math`，需谨慎，可能影响正确性
   * `-lineinfo` 仅 profiling 用，不一定用于最终

4. shape dispatch 变异：

   * 根据 `d` 选择不同 kernel config

5. 回退策略：

   * 连续失败 N 次，回到上一个成功 family
   * 超时前只做低风险微调

---

## 3.16 Best Candidate Manager

这是非常关键的工程模块。

**功能：**

1. 维护当前 best candidate。
2. 只有当 candidate 满足以下条件才更新：

   * 编译通过
   * 正确性通过
   * 综合 benchmark 更好
   * 多 shape 没有严重退化
3. 更新方式必须原子化：

```text
write candidate to optimized_lora.tmp
compile/test candidate
if better:
    copy candidate_i.cu -> optimized_lora.cu
    fsync / atomic rename
else:
    keep old optimized_lora.cu
```

4. 防止损坏最终文件：

   * 不直接覆盖 best
   * 保留 `optimized_lora.best.cu`
   * 保留 `optimized_lora.last_good.cu`

**设计原则：**

任何时刻 `optimized_lora.cu` 都应该是一个已经通过本地正确性测试的版本。

# 10. 关键工程保障

## 10.1 永远不让坏 candidate 覆盖 best



## 10.2 多 shape 验证，避免过拟合

至少测试：

```text
3584, 4096, 4608
```

更好再加：

```text
3712, 3840, 4352, 4480
```

因为 hidden evaluation 会在 `[3584, 4608]` 内选多个 d，作业明确要求不要 overfit 到一个固定 shape。

---

## 10.3 正确性优先级高于性能

更新 best 的条件应该是：

```text
compiled == true
correct == true
aggregate_speedup improved
no severe regression on any tested d
```

如果速度提升 2%，但某个 d 退化 20%，不应更新 best。

---


## 10.4 LLM 只负责写代码，不负责判断代码好坏

LLM 可能自信但错误，所以必须由 agent 自动验证。
