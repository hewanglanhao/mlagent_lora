# mlagent_lora 构建文档

本文档描述 `/workspace/mlagent_lora` 中 LoRA 算子优化 agent 系统的模块组成、模块职责、运行入口、整体工作流程以及主要输出产物。

## 1. 系统目标

本系统面向课程项目中的 LoRA 风格算子：

```text
Y = W @ X + A @ (B.T @ X)
```

其中：

- `W`: `d x d`
- `X`: `d x d`
- `A`: `d x 16`
- `B`: `d x 16`
- `d` 在 `[3584, 4608]` 范围内运行时变化
- 输入和输出均为 CUDA `float32` tensor

系统目标不是提交一个固定 kernel，而是构建一个可运行的多 agent 优化系统。系统会自动生成 CUDA/C++ 候选实现，编译、正确性测试、benchmark、profiling、分析瓶颈，并基于反馈继续生成或变异候选代码。运行期间始终维护当前可编译的最优版本 `optimized_lora.cu`。

## 2. 顶层目录结构

```text
/workspace/mlagent_lora
├── run.sh                         # 评测入口和环境初始化脚本
├── optimized_lora.cu              # 当前最终提交/评测使用的 CUDA 源文件
├── optimized_lora.best.cu         # 当前 best 的备份文件
├── optimized_lora.last_good.cu    # 最近通过验证的候选备份
├── agent/                         # 多 agent 主系统
├── agent/prompts/                 # LLM 各模块提示词
├── scripts/                       # 独立测试和 API smoke test 脚本
├── logs/                          # 每次运行单独日志目录
├── experiments/                   # 每次运行的候选代码、评测结果、LLM 调用记录
├── eval_results/                  # 本地评测脚本输出
└── doc/                           # 项目文档和环境变量文件
```

系统运行后还会向 `/workspace` 同步：

- `/workspace/optimized_lora.cu`
- `/workspace/output.md`

`output.md` 用于总结 agent 的搜索过程、候选对比、benchmark 结果、LLM 调用和最终选择。

## 3. 启动入口

系统的评测入口是：

```bash
bash /workspace/mlagent_lora/run.sh
```

`run.sh` 负责：

- 定位项目目录。
- 加载 `doc/环境变量.txt` 中的 API 和模型配置。
- 设置 Python、CUDA、PyTorch extension 编译相关环境变量。
- 启动后台同步任务，每隔数秒将仓库内 `optimized_lora.cu` 同步到 `/workspace/optimized_lora.cu`。
- 执行 `python3 -m agent.agent_framework` 启动 Supervisor。
- 若主流程异常退出，尝试执行 `--bootstrap-only` 生成保底可编译版本。
- 退出前再次同步最终 `optimized_lora.cu`。

关键环境变量包括：

| 变量 | 作用 |
| --- | --- |
| `OPENAI_API_KEY` / `API_KEY` | OpenAI 兼容 API 密钥 |
| `OPENAI_BASE_URL` / `BASE_URL` | OpenAI 兼容 API 地址 |
| `LLM_MODEL` / `BASE_MODEL` | 使用的 LLM 模型 |
| `MAX_OPT_TIME` | 单次优化运行内部时间上限，默认 `1700` 秒 |
| `STOP_MARGIN_SEC` | 接近超时前预留的停止余量 |
| `MIN_CANDIDATE_TIME_BUDGET_SEC` | 启动新候选前要求剩余的最小时间 |
| `LLM_TIMEOUT_SEC` | 单次 LLM 调用超时 |
| `MAX_CANDIDATES` | 最大候选迭代数量 |
| `BENCH_WARMUP` / `BENCH_ITERS` | benchmark warmup 和计时轮数 |
| `CORRECTNESS_RTOL` / `CORRECTNESS_ATOL` | 本地正确性测试容差 |
| `ENABLE_LLM_CLOSED_LOOP` | 是否启用严格 LLM 闭环 |
| `ASYNC_LLM_ADVISORY` | 是否后台执行非关键 LLM 审查/诊断 |

## 4. 核心模块

### 4.1 `agent.agent_framework`

这是系统主控模块，包含：

- `Supervisor`
- `CandidateRecord`
- `ConstraintSpecManager`
- `SearchSpacePlannerAgent`
- `BestCandidateManager`

`Supervisor` 是整个系统的调度中心。它负责初始化运行目录、日志系统、LLM 客户端、各功能 agent，并执行主优化循环。

主要职责：

- 管理时间预算和提前停止。
- 生成 baseline 候选并安装为初始 best。
- 调用环境检查、静态审查、编译、正确性测试、benchmark、profiling 和诊断模块。
- 在严格闭环模式下执行：

```text
LLM 规划/变异 -> LLM 生成代码 -> 静态审查 -> 编译 -> 正确性测试 -> benchmark/profile -> LLM 诊断 -> 下一轮
```

- 确保只有通过编译和正确性测试，并且满足 promotion 条件的候选才会更新 `optimized_lora.cu`。
- 写入 leaderboard、trace、最终报告 `output.md`。

`BestCandidateManager` 负责当前 best 的管理：

- 维护 `optimized_lora.cu`。
- 维护 `optimized_lora.best.cu`。
- 防止未验证候选覆盖已有 best。
- 在候选更优时执行原子复制。

### 4.2 `agent.spec`

该模块定义 LoRA 算子的固定约束。

核心类：

- `LoraSpec`

主要内容：

- operator 表达式。
- `d` 范围。
- rank 固定为 `16`。
- dtype 为 `float32`。
- 最终文件名 `optimized_lora.cu`。
- 正确性和 benchmark 使用的默认尺寸。

这些信息会被用于：

- LLM prompt 上下文。
- 静态审查约束。
- 本地 correctness/benchmark case 生成。
- 输出报告。

### 4.3 `agent.cuda_candidates`

该模块负责描述和生成候选实现。

核心类：

- `CandidateSpec`
- `CUDACandidateGenerator`

主要功能：

- 生成 deterministic baseline。
- 根据 LLM mutation 结果构造候选元数据。
- 生成候选 CUDA 文件。
- 提供候选去重 key，避免重复评估同类候选。
- 对候选名称做必要清理。

当前系统支持的候选方向包括：

- ATen reference baseline。
- pure cuBLAS three-SGEMM 风格实现。
- rank-16 update 相关变体。

### 4.4 `agent.llm_adapter`

该模块封装 LLM API 调用。

核心类：

- `LLMClient`
- `LLMResponse`

主要功能：

- 从环境变量读取 API key、base URL 和模型名。
- 优先使用 OpenAI SDK。
- 若 SDK 不可用，则使用 OpenAI-compatible HTTP fallback。
- 支持 `LLM_TIMEOUT_SEC` 控制单次请求超时。
- LLM 不可用时，系统仍可依靠 deterministic fallback 和本地验证继续运行。

系统中的 LLM 只负责规划、生成、诊断和建议；最终是否采用候选始终由本地编译、正确性测试和 benchmark 决定。

### 4.5 `agent.llm_agents`

该模块实现所有直接使用 LLM 的专职 agent。

主要类：

- `LLMCUDACandidateGeneratorAgent`
- `LLMStaticCodeReviewAgent`
- `PerformanceDiagnosisAgent`
- `OptimizationMutationAgent`

职责如下：

| Agent | 功能 |
| --- | --- |
| `LLMCUDACandidateGeneratorAgent` | 根据候选策略和历史结果生成完整单文件 CUDA/C++ extension |
| `LLMStaticCodeReviewAgent` | 对候选源码进行 LLM 静态审查，识别高风险实现和约束违反 |
| `PerformanceDiagnosisAgent` | 根据正确性、benchmark、profile 结果诊断瓶颈 |
| `OptimizationMutationAgent` | 根据当前 best 和历史实验提出下一轮候选策略 |

该模块还负责：

- 解析 LLM 返回的 JSON。
- 去除 Markdown code fence。
- 校验返回内容是否像可编译 CUDA extension。
- 在编译失败时触发 LLM repair codegen。

### 4.6 `agent/prompts`

该目录存放各 LLM 模块使用的 prompt。

主要文件：

```text
search_space_planner.system.md
search_space_planner.user.md
optimization_mutation.system.md
optimization_mutation.user.md
cuda_candidate_generator.system.md
cuda_candidate_generator.user.md
static_code_review.system.md
static_code_review.user.md
performance_diagnosis.system.md
performance_diagnosis.user.md
```

当前 prompt 重点引导 LLM 优先生成 pure cuBLAS three-SGEMM 风格候选：

- 不调用 ATen `mm` / `matmul` / `addmm` 完成核心计算。
- 不为 pure cuBLAS 策略手写 CUDA kernel。
- 通过 cuBLAS 的 column-major 解释实现 row-major 矩阵乘。
- 避免显式构造 `B.T`。
- 使用第三个 SGEMM 的 `beta = 1` 直接累加低秩项。

### 4.7 `agent.local_harness`

该模块是本地验证和测量核心。

主要类：

- `SyntheticDataGenerator`
- `EnvironmentInspectorAgent`
- `BuildCompileAgent`
- `CorrectnessTestAgent`
- `BenchmarkAgent`
- `ProfilingAgent`

职责如下：

| 模块 | 功能 |
| --- | --- |
| `SyntheticDataGenerator` | 在公开尺寸范围内生成 synthetic CUDA float32 输入 |
| `EnvironmentInspectorAgent` | 检查 PyTorch、CUDA、nvcc 和 GPU 信息 |
| `BuildCompileAgent` | 使用 `torch.utils.cpp_extension.load` 编译候选 `.cu` 文件 |
| `CorrectnessTestAgent` | 对比 `W @ X + A @ (B.T @ X)` PyTorch reference |
| `BenchmarkAgent` | 使用 CUDA event 计时，计算候选相对 PyTorch reference 的 speedup |
| `ProfilingAgent` | 汇总 profile/benchmark 证据，给出瓶颈标签 |

正确性测试会记录：

- 是否通过。
- 每个 d 的 `max_abs_err`。
- 每个 d 的 `rel_l2_err`。
- 失败 shape 列表。
- 使用的 `rtol` 和 `atol`。

benchmark 会记录：

- 候选 latency。
- PyTorch reference latency。
- 每个 d 的 speedup。
- aggregate speedup。
- warmup 和 iters 参数。

### 4.8 `agent.static_review`

该模块是本地静态审查器，不依赖 LLM。

主要类：

- `StaticCodeReviewAgent`
- `StaticReviewResult`

主要检查：

- 是否包含 `torch/extension.h`。
- 是否导出 `forward(W, X, A, B)`。
- 是否包含 `PYBIND11_MODULE`。
- 是否有明显的 hidden shape hardcode。
- 是否存在危险模式或不符合单文件约束的实现。

本地静态审查用于在编译前快速筛掉明显不合规候选。

### 4.9 `agent.background_tasks`

该模块封装后台任务。

主要功能：

- 在异步模式下后台执行 LLM 调用。
- 允许本地候选评估和 LLM advisory 并行。
- 记录后台任务耗时、返回值和异常。

严格闭环模式开启时，系统会优先保证诊断结果进入下一轮 codegen；异步 advisory 默认不会影响 gating 决策。

### 4.10 `agent.memory`

该模块管理实验数据写入。

主要功能：

- 原子写文本文件。
- 原子复制 best 文件。
- 创建每次运行的 `experiments/<run_id>` 目录。
- 写入 `run_metadata.json`。
- 写入 `agent_trace.jsonl`。
- 更新 `leaderboard.json`。

### 4.11 `agent.runtime_logging`

该模块负责运行日志初始化。

每次运行会生成独立日志目录：

```text
logs/<run_id>/agent.log
```

同时更新：

```text
logs/LATEST
```

日志中记录：

- run id。
- LLM 状态。
- 正确性阈值。
- pipeline 模式。
- 每个候选的编译、正确性、benchmark、promotion 决策。
- 每次 LLM 调用开始、成功、失败和耗时。
- 最终 best 和最终文件路径。

### 4.12 `agent.llm_call_logger`

该模块专门记录 LLM 调用。

每次运行会写入：

```text
experiments/<run_id>/llm_calls/
```

其中包括：

- `manifest.jsonl`: LLM 调用摘要。
- 每次调用的 system/user prompt 摘要。
- 响应状态、耗时、错误信息。
- JSON 或 CUDA code 解析结果。

该模块用于确认系统是否真的调用了 LLM，以及定位 timeout、空返回、解析失败等问题。

### 4.13 `agent.output_report`

该模块生成最终报告：

```text

```

报告包含：

- Run summary。
- 搜索过程统计。
- 候选对比表。
- 最佳候选详情。
- LLM 调用汇总。
- 环境信息。

Supervisor 写完报告后会读回校验，确认报告包含当前 `run_id`，避免出现日志显示写入但文件没有更新的情况。

### 5.1 流水线机制设计

系统采用“候选级流水线”而不是一次性生成最终代码。一次候选从策略产生到是否成为 best，会依次流过以下阶段：

```text
策略规划
  -> 候选生成
  -> 静态审查
  -> 编译
  -> 正确性测试
  -> benchmark
  -> profiling/诊断
  -> promotion 决策
  -> 下一轮策略
```

各阶段之间通过 `CandidateRecord` 和 `experiments/<run_id>/exp_<id>/` 中的 JSON 文件传递状态。每个候选都有独立目录，目录中保存：

- `candidate.cu`: 该轮生成的 CUDA/C++ 源码。
- `candidate_metadata.json`: 候选名称、family、父候选、生成来源等元数据。
- `static_review.json` / `llm_static_review.json`: 本地和 LLM 静态审查结果。
- `compile.json`: 编译是否成功、错误类型、编译耗时。
- `correctness.json`: 每个 shape 的正确性误差和是否通过。
- `benchmark.json`: 每个 shape 的 latency、reference latency 和 speedup。
- `profile.json` / `diagnosis.json`: 瓶颈证据和下一轮优化建议。

这种设计有几个目的：

1. **可恢复**：任一候选失败不会破坏当前 best，失败原因留在对应实验目录中。
2. **可追踪**：每个阶段的输入输出都落盘，便于复盘 LLM 是否真的调用、代码是否编译、性能是否提升。
3. **可闭环**：后续 LLM mutation 和 codegen 会读取历史候选、benchmark、profile 和 diagnosis，而不是盲目重新生成。
4. **可裁决**：LLM 可以建议和生成代码，但 promotion 只由本地编译、正确性和 benchmark 决定。

### 5.2 严格闭环流水线

默认启用 `ENABLE_LLM_CLOSED_LOOP=1` 时，系统使用严格闭环：

```text
LLM SearchSpacePlanner
  -> LLM OptimizationMutation
  -> LLM CUDACandidateGenerator
  -> StaticReview + LLMStaticReview
  -> BuildCompile
  -> CorrectnessTest
  -> Benchmark
  -> Profiling
  -> LLM PerformanceDiagnosis
  -> 下一轮 OptimizationMutation
```

严格闭环的特点是：当前候选的真实性能和诊断结果会进入下一轮 prompt。也就是说，系统不是简单地连续问 LLM 生成多个版本，而是让 LLM 看到：

- 哪些候选编译失败。
- 哪些候选正确性失败。
- 哪些 shape 变快或变慢。
- 当前 best 是哪个。
- 诊断模块认为瓶颈在哪里。
- 哪些方向已经尝试过、是否值得继续。

这使得系统能够形成如下反馈链：

```text
生成代码
  -> 编译/正确性/benchmark 暴露问题
  -> diagnosis 提炼瓶颈和失败原因
  -> mutation 将诊断转成下一轮改动
  -> codegen 生成新的 candidate.cu
```

如果候选编译失败，系统会在时间允许时触发 LLM repair codegen。repair prompt 会携带编译错误摘要、失败源码、历史实验和当前 best 摘要，尝试生成修复版候选。修复版仍然必须重新经历静态审查、编译、正确性、benchmark 和 promotion。

### 5.3 异步流水线与后台任务

系统也保留异步 LLM 机制，用于降低等待 LLM 的空转时间：

- `ENABLE_ASYNC_LLM`: 允许后台执行 LLM mutation/codegen。
- `ASYNC_LLM_CODEGEN`: 允许 CUDA codegen 后台运行。
- `ASYNC_LLM_ADVISORY`: 允许静态审查、诊断等非 gating LLM 调用后台运行。
- `ASYNC_LLM_IDLE_WAIT_SEC`: 没有本地候选可跑时，最多等待后台 LLM mutation 的时间。
- `ASYNC_LLM_STALE_RETRY_WAIT_SEC`: LLM 返回重复或无效建议后，再次等待建议的时间。

在严格闭环开启时，系统会优先保证“上一轮诊断进入下一轮生成”，因此关键 codegen 通常在前台执行。异步机制主要用于 advisory 或非关键任务，避免本地编译、正确性测试、benchmark 完全阻塞在 LLM 等待上。

### 5.4 Gating 和时间预算

流水线中真正的 gating 阶段是：

1. 本地静态审查。
2. LLM 静态审查风险记录。
3. 编译。
4. 正确性测试。
5. benchmark 和 promotion。

其中编译和正确性是硬门槛；benchmark 只有在候选正确时才执行。promotion 时还会比较当前候选和已有 best 的 aggregate speedup，只有满足提升条件才会覆盖 `optimized_lora.cu`。

时间预算也参与流水线调度。系统会在启动新候选前检查剩余时间：

- 若剩余时间不足 `MIN_CANDIDATE_TIME_BUDGET_SEC`，停止启动新候选。
- 若接近 `MAX_OPT_TIME` 或平台 30 分钟限制，进入收尾流程。
- 收尾时不强行等待长时间 LLM 任务，而是保留已经验证的 best。

因此，系统停止时的最终 `optimized_lora.cu` 应始终来自一个已通过本地验证的候选，而不是还在生成、还在编译或仅由 LLM 声称可用的代码。

## 6. 候选评估与 promotion 规则

每个候选按以下顺序验证：

1. 写入 `experiments/<run_id>/exp_<id>/candidate.cu`。
2. 本地静态审查。
3. LLM 静态审查。
4. 编译 PyTorch CUDA extension。
5. 正确性测试。
6. benchmark。
7. profiling/diagnosis。
8. promotion 决策。

promotion 的基本原则：

- 编译失败的候选不能 promotion。
- 正确性失败的候选不能 promotion。
- benchmark 不优于当前 best 的候选不能覆盖 best。
- baseline 只作为初始保底，不能在后续 bootstrap-only 场景中覆盖已有 best。
- LLM 静态审查 `pass=false` 会作为强 warning 写入日志和 promotion 决策记录。

当前运行中 `optimized_lora.cu` 始终指向已验证 best，因此即使后续候选失败，系统仍能返回一个可编译结果。

## 8. LLM 闭环设计

LLM 在系统中承担“建议者”和“代码生成者”角色，但不是最终裁判。

LLM 参与点：

- 搜索空间规划。
- 下一轮 mutation 策略。
- CUDA/C++ 候选代码生成。
- 编译失败后的代码修复。
- 静态风险审查。
- 性能瓶颈诊断。

本地系统承担裁判角色：

- 编译是否成功。
- 输出是否正确。
- speedup 是否真实。
- 是否允许覆盖 best。

这样设计可以避免 LLM 幻觉直接污染最终提交文件。


## 9. 失败与兜底机制

系统包含多层兜底：

- `run.sh` 启动后立即同步已有 `optimized_lora.cu`。
- Supervisor 启动后首先生成并安装 baseline。
- 每个候选单独存放在实验目录，不直接覆盖最终文件。
- 只有 promotion 后才更新 `optimized_lora.cu`。
- 主流程失败时，`run.sh` 会尝试 `--bootstrap-only`。
- 接近时间限制时停止启动新候选，保留当前 best。
- 输出报告写完后会校验当前 `run_id`。


## 10. 总结

本系统由 Supervisor 驱动多个专职 agent 协作完成 LoRA 算子优化。LLM 负责提出策略、生成代码和分析瓶颈，本地 harness 负责严格执行编译、正确性、benchmark 和 promotion。系统通过 per-run 日志、实验目录、LLM 调用记录、leaderboard 和最终 `output.md` 保存完整搜索轨迹，并始终维护可编译的 `optimized_lora.cu` 作为最终评测入口。
